## 2026-03-07 — Website Launch Execution Lessons

### What happened
- We spent significant effort on Mission Control tooling/UI while core website shipping progress lagged.
- Task orchestration was set up, but execution control was weak: multiple assignees timed out or returned invalid acknowledgements.
- Critical-path tasks were blocked by missing operational inputs (repo/task specs, target URLs, env/staging details).
- We reported motion (setup/dispatch) without enough hard output (published pages, finalized copy, QA-complete URLs).

### Root causes
1. **Outcome drift** — infrastructure work outweighed direct publish work.
2. **No strict start protocol** — tasks were assigned without enforced ACK+ETA+deliverable window.
3. **Dependency gating too late** — blockers were discovered after dispatch, not before.
4. **Insufficient fallback plan** — when agents timed out, reassignment/escalation was not immediate.

### Operating changes (effective now)
- **Output-first rule:** prioritize shipped pages/content over platform polish.
- **Strict ACK gate:** no task is considered started until owner returns `STARTING + ETA + first deliverable`.
- **Dependency-first dispatch:** include prerequisites and unblock decisions before assignment.
- **Timeout policy:** one failed/timeout acknowledgement triggers immediate reassignment or direct owner escalation.
- **Progress standard:** updates must include tangible artifacts (file paths, URLs, commits), not activity-only status.

### Website sprint guardrails
- Every 2-hour cycle must produce at least one concrete deliverable toward launch.
- Critical path is tracked explicitly: URL lock -> keyword validate -> briefs finalize -> template build -> QA -> publish.
- Any blocker older than 30 minutes is escalated with exact decision needed.
